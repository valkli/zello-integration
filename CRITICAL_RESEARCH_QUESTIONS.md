## Критические вопросы

### 1. TOC Byte и формат Opus пакетов
**Вопрос:** Почему TOC byte отличается между входящими и исходящими пакетами?

**Детали:**
- Входящий TOC: `0x18` = `00011000` (binary)
  - Config: 0 (bits 0-2)
  - Stereo: 1 (bit 3) - **СТЕРЕО**
  - Frame count: 1 (bits 4-6)
  
- Исходящий TOC: `0x58` = `01011000` (binary)
  - Config: 11 (bits 0-2) - **НЕПРАВИЛЬНО!**
  - Stereo: 0 (bit 3) - **МОНО**
  - Frame count: 0 (bits 1-0)

**Что проверить:**
- Правильно ли мы используем `opuslib.Encoder.encode()`?
- Нужны ли дополнительные параметры при вызове encode()?
- Правильно ли мы передаем frame_size (960 samples для 60ms)?
- Может быть, нужно использовать другой метод encode()?

**Где искать:**
- Документация opuslib: https://github.com/onbeep/opuslib
- RFC 6716 (Opus specification) - формат TOC byte
- Примеры использования opuslib для кодирования

### 4. Формат PCM данных для encoder
**Вопрос:** Правильно ли мы передаем PCM данные в encoder?

**Текущий подход:**
- TTS возвращает PCM 16-bit mono
- Конвертируем в стерео (дублируем канал)
- Разбиваем на кадры по 3840 bytes (960 samples × 2 channels × 2 bytes)
- Передаем в encoder.encode(pcm_data, frame_size=960)

**Что проверить:**
- Правильно ли мы конвертируем моно в стерео?
- Правильно ли мы разбиваем на кадры?
- Правильно ли мы передаем frame_size (960 для стерео или 1920)?
- Может быть, нужно передавать данные в другом формате (например, numpy array)?

**Где искать:**

### 6. Дополнительные параметры бинарного заголовка
**Вопрос:** Нужны ли дополнительные байты в бинарном заголовке кроме 9 байт?

**Текущая структура:**
- Byte 0: 0x01 (Packet Type)
- Bytes 1-4: Stream ID (Big Endian)
- Bytes 5-8: Packet ID (Big Endian)
- Bytes 9+: Opus payload
